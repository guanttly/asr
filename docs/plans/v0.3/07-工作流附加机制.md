# 07 — 工作流附加机制

## 1. 设计目标

- 所有开放 API（含兼容接口）默认走"该应用配置的默认工作流"。
- 调用方可在请求里显式指定 `workflow_id`，覆盖默认。
- 不传也不要求平台自带"硬编码"行为：必须有兜底默认。
- 工作流必须属于该应用的可见范围，避免跨应用越权使用未授权工作流。

## 2. 工作流可见性

平台引入"工作流所有权"维度（与现有 `OwnerType: system | user` 兼容）：

| OwnerType | 含义 | 谁可见 |
|---|---|---|
| `system` | 平台默认模板（v0.2 已有） | 所有应用、所有用户 |
| `user` | 管理员 / 业务用户私有（v0.2 已有） | 仅创建者 |
| `app` | **新增** —— 与某个开放应用绑定 | 该应用 + 平台管理员 |

实现：在 `workflows` 表上新增 `owner_app_id` 字段，与 `owner_user_id` 互斥。具体迁移见 [09-数据模型与迁移.md](09-数据模型与迁移.md)。

## 3. 默认工作流配置

在 `OpenApp.default_workflows` 中按能力维度配置，例如：

```json
{
  "asr.recognize": 12,
  "asr.stream": 12,
  "meeting.summary": 7,
  "skill.invoke": null
}
```

匹配优先级：

1. 请求体里 `workflow_id` 显式传入（且通过可见性校验）→ 使用之。
2. 应用 `default_workflows[capability]` 不为空 → 使用之。
3. 平台层 `system` 范畴中标记为该能力默认模板的工作流（标识：`workflow.is_capability_default && capability_key 匹配`）→ 使用之。
4. 仍无 → 跳过工作流处理（直接返回原始 ASR / Summary 结果）。

## 4. 校验规则

- **可见性**：`workflow_id` 不在 `system` / `app=<当前 app_id>` 范围 → `ERR_WORKFLOW_NOT_FOUND`。
- **节点匹配**：能力对工作流的节点有强约束（例如 meeting.summary 工作流必须含 `meeting_summary` 节点）→ 不匹配 `ERR_WORKFLOW_INVALID`。
- **状态**：工作流被禁用 / 删除 → `ERR_WORKFLOW_DISABLED`。
- **执行超时**：执行整体不得超过该能力的最大同步超时；超时返回 `ERR_WORKFLOW_TIMEOUT`，但已生成的 ASR / Summary 中间产物会包含在响应 `partial_data` 字段中（语义对齐 `ERR_ASR_PARTIAL`）。

## 5. 调用响应中的工作流字段

所有走工作流的响应统一附带：

```json
{
  "data": {
    "workflow_id": 12,
    "workflow_execution_id": 9012,
    "workflow_origin": "request" | "app_default" | "system_default",
    "post_processed_text": "..."   // 当工作流影响最终文本时
  }
}
```

`workflow_origin` 帮助对接方排错：是用了你传的、还是默认覆盖了。

## 6. 管理后台支持

应用详情页提供"默认工作流"区块：

- 列出该应用勾选的能力。
- 每条能力旁可下拉选择一个工作流：来源 = `system` 默认 + 该应用拥有的 `app` 工作流。
- 提供"测试一次"按钮，用平台管理员 token 即时跑一遍（不计入应用调用日志）。

## 7. 兼容接口的工作流处理

`old http.py` 的接口本身没有工作流概念。在兼容层：

- 兼容接口**没有 `app_id`**（无鉴权），无法读取 `OpenApp.default_workflows`；因此默认**不附加工作流**，行为与 0.2.x 完全一致。
- 旧接口不暴露 `workflow_id`，传入也忽略；对接方迁移到新接口才解锁"指定工作流"的能力。
- 若部署方希望兼容请求也跑某条 system 工作流，配置 `legacy.default_workflow_id_for_asr`、`legacy.default_workflow_id_for_meeting`（默认空）。
- 兼容请求不写 `open_call_logs.workflow_origin` 字段（与 [02-架构与授权设计.md](02-架构与授权设计.md#5-调用日志) 的隔离一致）。

## 8. 边界

- 工作流不可在请求中"内联定义"——必须先通过管理端创建，再用 ID 引用。
- 单次调用最多触发 1 个工作流（不支持链式 / 并行调用）。
- 工作流自身的复杂逻辑（条件分支等）属于工作流引擎职责（v0.2），v0.3 不扩展。
