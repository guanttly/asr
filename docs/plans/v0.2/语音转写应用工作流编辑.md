# 语音转写系统 v0.2 工作流改造计划

## TL;DR
将现有"硬编码后处理管道"改造为可视化编排的工作流系统。所有应用（实时转写、批量转写、会议纪要）均可绑定工作流。工作流由有序节点列表组成，每个节点（术语纠正、语气词过滤、LLM纠错、会议纪要生成等）可独立测试。管理员管理全局模板，用户可克隆自定义。

## 一、需求确认

| 决策项 | 选择 |
|--------|------|
| 工作流 UI | 有序列表编排（拖拽排序的节点卡片） |
| 工作流权限 | 混合模式：管理员管理全局模板 + 用户可克隆自定义 |
| 实时转写接入 | 录音结束后触发绑定的工作流做后处理 |
| 会议纪要 | 作为工作流末端可选输出节点 |
| LLM纠错 | 可配置后端（OpenAI 兼容接口，节点中配 URL+参数） |
| 节点测试 | 编辑器内置测试面板，输入样本文本查看节点处理结果 |

## 二、现状分析

### 当前问题
1. **目录混乱**：转写、实时识别、会议纪要功能散落在同一套 domain/application 中
2. **后处理硬编码**：`BatchMeetingProcessor` 中写死了 correction → summary → meeting 的管道
3. **无法灵活编排**：用户不能选择跳过/增加/调整处理步骤
4. **功能无法独立验证**：术语纠正、摘要生成等只能在完整管道中运行

### 需保留的核心基础设施
- ASR 引擎客户端（asrengine/）：批量+流式转写
- NLP 引擎（nlpengine/）：corrector + summarizer 作为节点实现基础
- WebSocket 通信：streaming + business event
- 网关路由：四服务架构

## 三、目标架构

### 3.1 功能模块拆分

```
应用层（独立目录）：
├── transcription/    # 批量转写（上传→ASR→触发工作流）
├── realtime/         # 实时语音识别（录音→流式ASR→结束后触发工作流）
├── meeting/          # 会议纪要（独立查看+作为工作流输出节点）
├── workflow/         # 工作流引擎（核心新增）
├── nlp/              # NLP 服务（纠错、摘要等，提供节点 API）
├── terminology/      # 术语库管理
└── user/             # 用户管理
```

### 3.2 工作流领域模型

```
Workflow（工作流模板）
├── ID, Name, Description
├── OwnerType: "system" | "user"
├── OwnerID: 管理员ID 或 用户ID
├── SourceID: 克隆来源ID（可选）
├── IsPublished: bool
├── Nodes: []WorkflowNode（有序）
└── CreatedAt, UpdatedAt

WorkflowNode（工作流节点）
├── ID, WorkflowID
├── NodeType: 枚举（见下）
├── Position: int（排序序号）
├── Config: JSON（节点特定配置）
└── Enabled: bool

NodeType 枚举：
- term_correction    # 术语纠正（选择 DictID）
- filler_filter      # 语气词过滤（配置过滤词表）
- llm_correction     # LLM 纠错（配置 endpoint/model/prompt）
- speaker_diarize    # 说话人分离
- meeting_summary    # 会议纪要生成（配置 prompt 模板）
- custom_regex       # 自定义正则替换（pattern + replacement）

WorkflowExecution（工作流执行记录）
├── ID, WorkflowID, TriggerType, TriggerID
├── InputText, FinalText
├── Status: pending → running → completed | failed
├── NodeResults: []NodeExecutionResult
└── StartedAt, CompletedAt

NodeExecutionResult（单节点执行结果）
├── ID, ExecutionID, NodeID, NodeType
├── InputText, OutputText
├── Status: success | failed | skipped
├── Detail: JSON（修正记录、错误信息等）
├── Duration: ms
└── ExecutedAt
```

### 3.3 工作流引擎设计

引擎是一个有序管道执行器：

```
WorkflowEngine
├── RegisterNodeHandler(nodeType, handler)
├── Execute(ctx, workflowID, inputText, triggerMeta) → ExecutionResult
└── TestNode(ctx, nodeType, config, inputText) → NodeTestResult

NodeHandler interface {
  Execute(ctx, config JSON, inputText) → (outputText, detail, error)
  Validate(config JSON) → error
}
```

**节点处理器实现**：
- `TermCorrectionHandler` → 复用现有 `nlpengine.Corrector`
- `FillerFilterHandler` → 新增：基于词表的语气词/停顿词过滤
- `LLMCorrectionHandler` → 新增：调用 OpenAI 兼容接口做文本纠错
- `SpeakerDiarizeHandler` → 复用现有 `diarization.Client`
- `MeetingSummaryHandler` → 复用/升级现有 `nlpengine.Summarizer`
- `CustomRegexHandler` → 新增：用户自定义正则替换

### 3.4 触发机制

| 应用 | 触发时机 | 工作流选择 |
|------|---------|-----------|
| 批量转写 | ASR 引擎返回原始文本后 | 创建任务时选择工作流 |
| 实时转写 | 用户点击"停止录音"后 | 开始录音前选择工作流 |
| 手动执行 | 工作流编辑器中测试 | 当前编辑的工作流 |

替代现有 `BatchMeetingProcessor` 的硬编码管道。

### Phase 4: 新增节点实现

**步骤 4.1** 语气词过滤节点 —— *可独立开发*
- 后端：`handler_filler_filter.go`
  - Config: `{ filter_words: ["嗯","啊","那个","就是说",...], custom_words: [] }`
  - 逻辑：分词 → 移除匹配词 → 清理多余空格/标点
- 前端：节点配置表单（预设词表 + 自定义输入）

**步骤 4.2** LLM 纠错节点 —— *可独立开发*
- 后端：`handler_llm_correction.go`
  - Config: `{ endpoint: "http://...", model: "qwen3-4b", api_key: "", prompt_template: "...", temperature: 0.3, max_tokens: 4096 }`
  - 逻辑：构造 OpenAI ChatCompletion 请求 → 发送 → 提取纠错后文本
- 前端：节点配置表单（endpoint/model/prompt 模板编辑器/参数调节）

**步骤 4.3** 自定义正则节点 —— *可独立开发*
- 后端：`handler_custom_regex.go`
  - Config: `{ rules: [{ pattern: "regex", replacement: "text", enabled: true }] }`
- 前端：规则列表编辑器（pattern + replacement + 测试）

**步骤 4.4** 升级会议纪要节点 —— *可独立开发*
- 后端：升级 `handler_meeting_summary.go`
  - Config: `{ endpoint: "...", model: "...", prompt_template: "...", output_format: "markdown" }`
  - 替代现有占位符 Summarizer，接入真实 LLM
  - 执行后自动创建/更新 Meeting 记录
- 前端：配置 prompt 模板 + 输出格式

**步骤 4.5** 说话人分离节点 —— *可独立开发*
- 后端：`handler_speaker_diarize.go`
  - Config: `{ service_url: "http://pyannote:8001" }`
  - 注意：该节点需要音频 URL 而非文本输入，需特殊处理
  - 输出：带说话人标签的分段文本

## 八、验证方案

1. **单元测试**：每个 NodeHandler 的 Execute + Validate 方法单测
2. **集成测试**：WorkflowEngine 串联多节点执行、状态流转、错误处理
3. **API 测试**：工作流 CRUD + test-node + execute 接口
4. **前端手动验证**：
   - 创建工作流 → 添加节点 → 拖拽排序 → 配置参数 → 单节点测试 → 全流程测试
   - 批量转写选择工作流 → 查看执行详情
   - 实时转写选择工作流 → 停止后查看后处理结果
   - 克隆系统模板 → 修改 → 使用
5. **兼容性验证**：无 workflowID 的旧任务仍走原 postprocessor 逻辑

## 九、排除范围

- 不改变四服务架构（gateway + asr-api + admin-api + nlp-api）
