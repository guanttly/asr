# 前端收口优化计划

## 目标

在当前“应用 / 工作流 / 术语库 / 系统管理”信息架构基础上，把导航、应用配置、工作流联动、页面文案、代码复用和验证流程一次性收口，避免继续出现“结构已调整，但页面行为、文案和技术实现仍然分散”的半成品状态。

## 当前状态总结

### 已完成
- 导航已从“转写业务”拆为“应用 / 工作流 / 术语库 / 系统管理”。
- 已新增应用配置页，用于集中绑定实时、批量、会议纪要的默认工作流。
- 实时、批量、会议页面已去掉大部分页面内工作流选择器，改为消费应用配置。
- 前端构建当前可通过。

### 还未真正收干净的点
- 应用配置仍然只保存在本地 `localStorage`，不是用户级、服务端级配置。
- 工作流类型、工作流列表加载、绑定提示逻辑在多个页面重复实现。
- 页面文案虽然已改方向，但提示口径还不完全统一。
- 仍有遗留页面 `frontend/src/pages/transcription/index.vue` 未接入当前路由，属于死代码候选。
- 前端没有自动化测试，当前收口主要依赖人工回归。

## 一次性收口范围

### P0：本轮必须完成

#### 1. 信息架构彻底收口
- 保持一级导航稳定为：总览、应用、工作流、术语库、系统管理。
- 应用二级菜单固定为：实时语音识别、批量转写、会议纪要。
- 工作流二级菜单固定为：工作流管理、应用配置。
- 工作流、术语库、系统管理均保持独立一级，不再回退混放。
- 将术语规则入口纳入术语库子导航，避免“术语库管理”和“纠错规则”路由关系割裂。
- 清理未使用或已被替代的页面与路由，优先处理 `frontend/src/pages/transcription/index.vue`。

验收标准：
- 菜单命名、页面标题、路由描述全部一致。
- 不存在“页面逻辑已迁移，但旧页面仍留在仓库中”的悬空入口。

#### 2. 应用配置能力真正闭环
- 抽出统一的“应用默认工作流”数据模型，不再在多个页面手写类型。
- 抽出统一的工作流列表加载能力：按 `workflow_type` 获取、过滤 legacy、映射预览结构。
- 抽出统一的绑定校验能力：未配置、已下线、legacy、类型不匹配都走同一套提示逻辑。
- 页面只消费“当前应用配置状态”，不自己拼装业务规则。
- 为应用配置页增加“重置配置”“跳转管理工作流”“跳转对应应用”这三类明确动作。

验收标准：
- 实时、批量、会议页对默认工作流的展示和提示完全一致。
- 不再出现 4 份以上重复的 `WorkflowOption` 类型和映射逻辑。

#### 3. 业务页面文案和交互统一
- 实时页、批量页、会议页统一使用“默认工作流”口径，不再混用“选择工作流”“绑定工作流”“后处理工作流”。
- 所有页面统一区分两个概念：
  - 当前应用默认工作流
  - 当前任务/会议实际绑定的工作流
- 详情页必须明确说明：任务是在当时使用哪条工作流执行的，而不是只展示现在的默认配置。
- 页面中的“去应用配置”“去工作流管理”跳转文案统一。

验收标准：
- 用户不会误以为“改了应用配置就会影响历史任务”。
- 所有提示文案都能回答这三个问题：现在默认用哪个、这次实际用了哪个、配置失效后该去哪修。

#### 4. 视觉与导航体验收口
- 收起态菜单补 Tooltip，保证只看图标时仍能识别菜单含义。
- 统一侧栏一级/二级图标尺寸、间距、选中态和折叠态表现。
- 顶部标题区替换当前占位式图标，改成和侧栏同一风格的 SVG 系统。
- 应用配置页、工作流页、应用执行页统一卡片密度与按钮布局，不再出现“有的页像后台，有的页像临时调试面板”的割裂感。

验收标准：
- 折叠态侧栏可用，不依赖记忆猜图标。
- 页面视觉层级统一，没有明显的占位或临时样式。

#### 5. 回归验证流程补齐
- 增加至少一份前端人工回归清单，覆盖：
  - 应用配置变更
  - 实时转写保存
  - 批量上传 / URL 提交
  - 会议创建 / 摘要生成
  - 历史详情页查看执行结果
  - legacy / 缺失工作流提示
- 在没有自动化测试前，构建与人工回归必须绑定执行。

验收标准：
- 每次改动后都能按固定 checklist 验证，而不是靠临场想流程。

### P1：紧随其后

#### 6. 配置持久化从本地升级到服务端
- 当前 `workflowBindings` 仅存本地浏览器，存在换设备丢失、多人共用账号不一致的问题。
- 应新增服务端“应用默认工作流配置”接口，至少做到用户级持久化。
- 后端应校验配置合法性，避免前端绑定已下线工作流后只有 UI 警告，没有正式约束。

验收标准：
- 同一账号跨浏览器或跨机器登录后，应用配置一致。

#### 7. 工作流中心与应用配置打通
- 应用配置页支持“只看兼容工作流”。
- 从应用配置页一键跳到“新建对应类型工作流”。
- 工作流列表中明确展示“当前被哪些应用配置引用”。
- 对 legacy 工作流增加更清晰的升级入口，而不仅仅是警告文案。

验收标准：
- 用户能够从“应用配置缺失”顺畅走到“创建/选择兼容工作流”。

#### 8. 代码复用与边界整理
- 提取公共类型：工作流预览项、工作流列表项、绑定状态。
- 提取公共 composable：
  - `useWorkflowCatalog(workflowType)`
  - `useWorkflowBindingStatus(bindingKey, options)`
- 让页面只处理 UI，不重复承载工作流查询、映射、校验、提示生成。
- 将导航 SVG 图标提取到独立模块，避免 `DefaultLayout.vue` 继续膨胀。

验收标准：
- 新增一个“应用型页面”时，不需要再复制一套工作流类型和列表处理逻辑。

### P2：增强项

#### 9. 自动化测试基线
- 当前前端没有测试目录，也没有测试脚本。
- 建议最小化引入：
  - 单元测试：Vitest
  - 关键流程冒烟：Playwright 或最少一套脚本化手工流程
- 首批只覆盖最关键的 3 条链路：
  - 修改应用配置
  - 发起批量任务
  - 会议摘要重新生成

#### 10. 文档与发布说明
- 更新 v0.2 文档，明确新的“应用配置 -> 应用执行 -> 工作流结果”链路。
- 增加一页面向内部使用者的操作说明：
  - 如何配置默认工作流
  - 历史任务为什么不受默认配置变更影响
  - legacy 工作流如何迁移

## 推荐实施顺序

### 第 1 阶段：结构清理
- 清理死代码和旧页面入口。
- 补齐术语库子导航。
- 统一路由 meta、页面标题、菜单命名。

### 第 2 阶段：应用配置能力抽象
- 提取 workflow 相关公共类型。
- 提取工作流加载与绑定状态 composable。
- 把实时、批量、会议三页改成完全消费抽象层。

### 第 3 阶段：页面交互统一
- 统一默认工作流状态卡。
- 统一跳转入口、空状态、错误提示。
- 统一详情页“默认配置”和“实际执行”区分。

### 第 4 阶段：视觉与折叠态收尾
- 菜单折叠 Tooltip。
- 顶栏图标和侧栏图标统一。
- 卡片、按钮、提示条样式统一。

### 第 5 阶段：验证与文档
- 建立人工回归 checklist。
- 更新 v0.2 方案文档和操作说明。
- 在条件允许时补最小自动化测试基线。

## 定义完成（Definition of Done）
- 导航结构、路由、页面标题、页面文案完全一致。
- 应用配置、实时、批量、会议页面不再重复实现工作流绑定逻辑。
- 默认工作流与历史任务实际工作流的边界清晰可见。
- 不存在未使用旧页面、旧入口、旧文案残留。
- 折叠态导航、空状态、失效状态都有明确反馈。
- `pnpm build` 和 `pnpm lint` 可通过。
- 有一份实际可执行的人工回归清单。

- 删除或确认归档 `frontend/src/pages/transcription/index.vue`。
- 提取公共 workflow 类型与 `useWorkflowCatalog`。
- 提取公共 `useWorkflowBindingStatus`。
- 给侧栏折叠态补 Tooltip，并统一顶栏 SVG。
- 把术语规则并入术语库导航关系。
- 补一份前端回归检查文档。
