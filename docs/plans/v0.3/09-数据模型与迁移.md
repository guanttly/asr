# 09 — 数据模型与迁移

新增的迁移文件统一放在 `backend/migrations/`，按现有 `00X_*.sql` 命名规则编号。本节列出 v0.3 必须新增的 schema 与对既有表的最小改动。

## 1. 新增表

### 1.1 `011_open_apps.sql` — 开放应用

```sql
CREATE TABLE IF NOT EXISTS open_apps (
  id              BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  app_id          VARCHAR(48)  NOT NULL UNIQUE,
  name            VARCHAR(64)  NOT NULL UNIQUE,
  description     VARCHAR(512) NULL,
  app_secret_hash VARCHAR(255) NOT NULL,         -- Argon2id
  secret_version  INT UNSIGNED NOT NULL DEFAULT 1,
  status          ENUM('active','disabled','revoked') NOT NULL DEFAULT 'active',
  owner_user_id   BIGINT UNSIGNED NOT NULL,
  rate_limit_per_sec INT UNSIGNED NOT NULL DEFAULT 30,
  callback_whitelist JSON NULL,                  -- ["https://a.com",...]
  default_workflows  JSON NULL,                  -- { "asr.recognize": 12, ... }
  meta            JSON NULL,
  created_at      DATETIME NOT NULL,
  updated_at      DATETIME NOT NULL,
  PRIMARY KEY (id),
  KEY idx_owner (owner_user_id)
) DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_as_ci;
```

### 1.2 `012_open_app_capabilities.sql` — 应用能力授权

```sql
CREATE TABLE IF NOT EXISTS open_app_capabilities (
  id          BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  app_id      BIGINT UNSIGNED NOT NULL,
  capability  VARCHAR(64)  NOT NULL,
  enabled     TINYINT(1)   NOT NULL DEFAULT 1,
  created_at  DATETIME NOT NULL,
  updated_at  DATETIME NOT NULL,
  PRIMARY KEY (id),
  UNIQUE KEY uk_app_cap (app_id, capability),
  CONSTRAINT fk_oac_app FOREIGN KEY (app_id) REFERENCES open_apps(id) ON DELETE CASCADE
);
```

> 也可考虑直接用 `open_apps.allowed_caps JSON`；选择独立表的好处是后续要加配额 / 时窗细分时不必再次迁移。

### 1.3 `013_open_skills.sql` — Skill 注册中心

```sql
CREATE TABLE IF NOT EXISTS open_skills (
  id                BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  skill_uid         VARCHAR(48) NOT NULL UNIQUE, -- skl_xxxx
  app_id            BIGINT UNSIGNED NOT NULL,
  name              VARCHAR(64) NOT NULL,
  display_name      VARCHAR(128) NOT NULL,
  description       VARCHAR(512) NULL,
  intent_patterns   JSON NOT NULL,
  parameters_schema JSON NULL,
  callback_url      VARCHAR(512) NOT NULL,
  callback_timeout_ms INT UNSIGNED NOT NULL DEFAULT 3000,
  enabled           TINYINT(1) NOT NULL DEFAULT 1,
  consecutive_failures INT UNSIGNED NOT NULL DEFAULT 0,
  last_failure_at   DATETIME NULL,
  created_at        DATETIME NOT NULL,
  updated_at        DATETIME NOT NULL,
  PRIMARY KEY (id),
  UNIQUE KEY uk_app_name (app_id, name),
  CONSTRAINT fk_skill_app FOREIGN KEY (app_id) REFERENCES open_apps(id) ON DELETE CASCADE
);
```

### 1.4 `014_open_call_logs.sql` — 调用日志

```sql
CREATE TABLE IF NOT EXISTS open_call_logs (
  id          BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  request_id  VARCHAR(48) NOT NULL,
  app_id      BIGINT UNSIGNED NOT NULL,
  capability  VARCHAR(64) NOT NULL,
  route       VARCHAR(255) NOT NULL,
  http_status SMALLINT UNSIGNED NOT NULL,
  err_code    VARCHAR(64) NULL,
  latency_ms  INT UNSIGNED NOT NULL,
  ip          VARCHAR(64) NULL,
  user_agent  VARCHAR(255) NULL,
  body_ref    VARCHAR(255) NULL,                 -- 指向 runtime/openapi-logs/...
  created_at  DATETIME NOT NULL,
  PRIMARY KEY (id),
  KEY idx_app_time (app_id, created_at),
  KEY idx_capability_time (capability, created_at),
  KEY idx_request (request_id)
);
```

### 1.5 `015_skill_invocations.sql` — Skill 回调记录

```sql
CREATE TABLE IF NOT EXISTS skill_invocations (
  id              BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  skill_id        BIGINT UNSIGNED NOT NULL,
  app_id          BIGINT UNSIGNED NOT NULL,
  request_id      VARCHAR(48) NOT NULL,
  matched_pattern VARCHAR(255) NULL,
  utterance       TEXT NULL,
  parameters      JSON NULL,
  status          ENUM('success','failed','timeout','signed_rejected') NOT NULL,
  http_status     SMALLINT UNSIGNED NULL,
  latency_ms      INT UNSIGNED NULL,
  error_message   VARCHAR(512) NULL,
  created_at      DATETIME NOT NULL,
  PRIMARY KEY (id),
  KEY idx_skill_time (skill_id, created_at),
  KEY idx_app_time (app_id, created_at),
  CONSTRAINT fk_inv_skill FOREIGN KEY (skill_id) REFERENCES open_skills(id) ON DELETE CASCADE
);
```

## 2. 既有表改动

### 2.1 `workflows` 增加 app 归属

```sql
ALTER TABLE workflows
  ADD COLUMN owner_app_id BIGINT UNSIGNED NULL AFTER owner_user_id,
  ADD KEY idx_owner_app (owner_app_id),
  ADD CONSTRAINT chk_owner_exclusive
    CHECK (
      (owner_user_id IS NOT NULL AND owner_app_id IS NULL) OR
      (owner_user_id IS NULL     AND owner_app_id IS NOT NULL) OR
      (owner_user_id IS NULL     AND owner_app_id IS NULL)  -- system 模板
    );
```

> `owner_user_id` 与 `owner_app_id` 互斥；都为空表示 system 模板（与 `OwnerType` 字段共同判定，保持向后兼容）。

### 2.2 `workflows` 标记能力默认

```sql
ALTER TABLE workflows
  ADD COLUMN is_capability_default TINYINT(1) NOT NULL DEFAULT 0 AFTER owner_app_id,
  ADD COLUMN capability_key       VARCHAR(64) NULL              AFTER is_capability_default,
  ADD KEY idx_capability_default (capability_key, is_capability_default);
```

约束：同一 `capability_key` 下 `is_capability_default=1` 的 system 工作流至多一个；通过应用层校验（CHECK 约束在 MySQL 8 中支持但维护成本高，使用唯一过滤索引即可）：

```sql
CREATE UNIQUE INDEX uk_capability_default
  ON workflows (capability_key, is_capability_default)
  WHERE is_capability_default = 1 AND owner_user_id IS NULL AND owner_app_id IS NULL;
```

> MySQL 8 不直接支持过滤唯一索引，可改为应用层服务保证；本规划记录意图，由实现阶段二选一。

### 2.3 `voice_command_dicts` 增加 app 归属（可选）

如果选择"per-app skill 词典"路线：

```sql
ALTER TABLE voice_command_dicts
  ADD COLUMN owner_app_id BIGINT UNSIGNED NULL AFTER owner_user_id,
  ADD KEY idx_owner_app (owner_app_id);
```

否则 skill 命中逻辑直接读取 `open_skills.intent_patterns`，不与既有指令字典共表。本计划默认选择**后者**（不改 `voice_command_dicts`），保持桌面端语音指令独立。

## 3. 数据流

- 创建应用 → 写 `open_apps` + `open_app_capabilities`。
- rotate-secret → 更新 `app_secret_hash` 与 `secret_version+1`，旧 token 自动失效（中间件比对版本）。
- 调用 → OpenAuth 中间件命中 → 写 `open_call_logs`（路由处统一中间件落库，handler 不感知）。
- skill 回调 → asr-api 在意图命中后查 `open_skills` → 发起 webhook → 写 `skill_invocations`。

## 4. 索引与体积估算

- `open_call_logs` 是高写入表；按 `app_id + created_at` 索引最多承载 5k QPS 的写入（私有化场景一般在 50 QPS 以内）。
- 30 天保留期内，按 100 万行 / 天估算约 30 GB / 月 — 远低于 v0.3 范围所需，无须分表。

## 5. 删除策略

- 删除 `open_apps` 时级联删除 `open_app_capabilities` / `open_skills` / `skill_invocations`。
- 不级联 `open_call_logs`（保留为审计）；改为软引用（`app_id` 仍保留，但 `open_apps` 行已不存在）；查询页面在加载时容忍。
- 删除工作流时若被某应用作为默认引用，必须前置校验（与字典删除保护一致）。
