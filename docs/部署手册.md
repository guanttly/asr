# 巨鲨语音转写系统部署手册

本文档面向现场运维人员，说明交付包如何安装、升级、修改访问 IP、调整配置以及排查常见问题。除非研发或交付人员特别说明，运维只需要按本文档执行，不需要了解系统内部实现。

## 1. 部署前确认

### 1.1 交付物说明

正式交付一般是以下三类安装包之一。`.run` 文件和同名 `.run.part001`、`.run.part002` 等分包必须放在同一个目录，文件名不要改。

| 交付物 | 用途 | 典型部署方式 |
| --- | --- | --- |
| `jusha-asr-all-<版本>.run` | 全量包，包含业务服务、ASR 模型服务、3D-Speaker 人声服务 | 一台服务器部署完整系统，优先使用 |
| `jusha-asr-business-<版本>.run` | 业务服务包，包含 Web、后台服务、MySQL、Nginx、客户端下载页 | 单独部署业务服务，模型服务已在本机或其他服务器 |
| `jusha-asr-models-<版本>.run` | 模型服务组合包，包含 ASR 推理服务和 3D-Speaker 人声服务 | 单独部署模型服务，供业务服务调用 |

如果交付目录里只有 `jusha-asr-business` 和 `jusha-asr-models` 两套包，说明现场需要按“分离部署”安装。如果有 `jusha-asr-all`，通常直接按“单机全量部署”安装。

### 1.2 服务器要求

| 项目 | 要求 |
| --- | --- |
| 操作系统 | Linux 服务器，推荐 Ubuntu 22.04/24.04 或同等级发行版 |
| 权限 | 具备 `root` 或 `sudo` 权限，当前用户可以执行 Docker 命令 |
| Docker | 已安装 Docker，并安装 `docker compose` 插件或 `docker-compose` |
| GPU | 部署 `jusha-asr-all` 或 `jusha-asr-models` 时需要 NVIDIA GPU、驱动和 `nvidia-container-toolkit` |
| 端口 | 默认使用 `9855`、`9856`、`9851`、`9852`，安装前确认未被占用 |
| 磁盘 | 标准版最低 200G SSD，推荐 500G SSD；模型包较大，建议放在数据盘 |

硬件规格按合同或项目交付清单确认。标准版最低 CPU 8 核、内存 16G、RTX 3090 同等算力；高级版最低 CPU 16 核、内存 32G、A10 同等算力。

### 1.3 需要安装的软件包

安装业务或模型交付包前，服务器至少需要以下软件。交付包不会自动安装 Docker、显卡驱动或系统依赖。

| 软件 | 说明 |
| --- | --- |
| `bash`、`sh`、`tar`、`gzip`、`coreutils` | Linux 常用基础命令，一般系统自带 |
| Docker Engine | 用于运行业务服务和模型服务容器 |
| Docker Compose | 推荐 Compose V2，即 `docker compose`；旧环境可使用 `docker-compose` |
| NVIDIA Driver | 仅模型服务器需要，安装后 `nvidia-smi` 应能看到 GPU |
| NVIDIA Container Toolkit | 仅模型服务器需要，用于容器访问 GPU |
| `curl` 或浏览器 | 用于安装后做连通性检查 |

安装前建议执行以下检查：

```bash
docker --version
docker compose version || docker-compose --version
docker info
nvidia-smi
```

没有 GPU 的业务服务器可以跳过 `nvidia-smi`。

## 2. 默认端口和访问地址

| 服务 | 默认端口 | 用途 |
| --- | --- | --- |
| 业务 HTTP | `9855` | 桌面客户端或普通 HTTP 访问 |
| 业务 HTTPS | `9856` | 浏览器访问、下载页、麦克风实时录音推荐入口 |
| ASR 推理服务 | `9851` | 模型识别服务，业务服务调用 |
| 3D-Speaker 人声服务 | `9852` | 说话人分离、声纹等能力，业务服务调用 |

安装完成后的常用访问地址如下，将 `<服务器IP>` 替换为现场服务器 IP：

```text
浏览器登录页: https://<服务器IP>:9856/login
客户端下载页: https://<服务器IP>:9856/downloads
桌面客户端地址: http://<服务器IP>:9855
```

HTTPS 默认使用自签名证书。首次访问时浏览器可能提示证书不受信任，按现场安全要求导入证书或继续访问即可。证书文件在业务包目录的 `runtime/certs/tls.crt`。

## 3. 安装目录约定

建议统一安装在 `/opt/jusha-asr` 或数据盘目录，例如 `/data/jusha-asr`。下文以 `/opt/jusha-asr` 为例。

```bash
sudo mkdir -p /opt/jusha-asr
sudo chown -R $USER:$USER /opt/jusha-asr
cd /opt/jusha-asr
```

请将交付的 `.run` 和所有 `.run.partNNN` 文件复制到该目录。不要只复制 `.run` 文件，否则安装时会提示找不到分包文件。

## 4. 部署方式一：单机全量部署

适用场景：交付物是 `jusha-asr-all-<版本>.run`，并且业务服务和模型服务都部署在同一台服务器。

### 4.1 安装

```bash
cd /opt/jusha-asr
ls jusha-asr-all-*.run*
sh jusha-asr-all-<版本>.run
```

安装脚本会自动完成以下工作：

- 解包全量包。
- 安装 ASR 推理服务。
- 安装 3D-Speaker 人声服务。
- 安装业务服务。
- 加载离线 Docker 镜像并启动容器。
- 自动选择 Docker 内部网络，避免和现场内网、VPN、已有 Docker 网络冲突。
- 输出最终访问地址和证书提示。

安装成功后目录通常如下：

```text
/opt/jusha-asr/jusha-asr-all/
├── jusha-asr-models/
│   ├── jusha-asr-asr/
│   └── jusha-asr-speaker/
└── jusha-asr-business/
```

### 4.2 验证

```bash
docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}'
curl -k https://127.0.0.1:9856/healthz
curl http://127.0.0.1:9852/api/v1/health
```

然后在浏览器访问：

```text
https://<服务器IP>:9856/login
https://<服务器IP>:9856/downloads
```

默认管理员账号以交付包配置为准；未特别定制时通常为 `admin`，默认密码以项目交付清单为准。首次登录后请及时修改密码。

## 5. 部署方式二：业务服务和模型服务分离部署

适用场景：模型服务部署在 GPU 服务器，业务服务部署在另一台服务器，或者先部署模型服务再部署业务服务。

### 5.1 在模型服务器安装 `jusha-asr-models`

```bash
cd /opt/jusha-asr
ls jusha-asr-models-*.run*
sh jusha-asr-models-<版本>.run
```

安装完成后检查容器和人声服务健康状态：

```bash
docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}'
curl http://127.0.0.1:9852/api/v1/health
```

模型服务器需要对业务服务器开放以下端口：

```text
9851  ASR 推理服务
9852  3D-Speaker 人声服务
```

### 5.2 在业务服务器安装 `jusha-asr-business`

```bash
cd /opt/jusha-asr
ls jusha-asr-business-*.run*
sh jusha-asr-business-<版本>.run
```

如果交付包打包时已经写入模型服务器地址，安装完成后可直接验证。

如果安装后需要手工指定模型服务器地址，编辑业务包目录下的 `.env`：

```bash
cd /opt/jusha-asr/jusha-asr-business
cp .env .env.bak.$(date +%Y%m%d%H%M%S)
vi .env
```

将以下配置改为模型服务器的实际 IP：

```env
ASR_SERVICES_ASR=http://<模型服务器IP>:9851
ASR_SERVICES_SPEAKER_SERVICE_URL=http://<模型服务器IP>:9852
```

保存后执行升级命令，让配置重新生效：

```bash
sh install.sh upgrade
```

如果业务服务和模型服务在同一台服务器，但因为特殊原因不能使用默认 Docker 内部服务名，也可以改成宿主机端口：

```env
ASR_SERVICES_ASR=http://host.docker.internal:9851
ASR_SERVICES_SPEAKER_SERVICE_URL=http://host.docker.internal:9852
```

### 5.3 验证

在业务服务器执行：

```bash
docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}'
curl -k https://127.0.0.1:9856/healthz
```

在浏览器访问：

```text
https://<业务服务器IP>:9856/login
https://<业务服务器IP>:9856/downloads
```

登录后上传一段测试音频，确认识别任务可以完成。

## 6. 部署方式三：仅安装业务服务包

适用场景：现场已经存在可用 ASR 和 3D-Speaker 服务，只需要部署 Web、后台服务、数据库和客户端下载页。

安装命令：

```bash
cd /opt/jusha-asr
sh jusha-asr-business-<版本>.run
```

安装后编辑 `.env`，指定已有模型服务地址：

```bash
cd /opt/jusha-asr/jusha-asr-business
vi .env
```

```env
ASR_SERVICES_ASR=http://<ASR服务IP或域名>:<端口>
ASR_SERVICES_SPEAKER_SERVICE_URL=http://<人声服务IP或域名>:<端口>
```

保存后执行：

```bash
sh install.sh upgrade
```

注意：`ASR_BOOTSTRAP_ADMIN_PASSWORD` 只在首次初始化数据库时创建管理员使用。如果数据库已经初始化，之后只改 `.env` 里的管理员密码不会覆盖已有账号密码。

## 7. 部署方式四：仅安装模型服务包

适用场景：该服务器只提供 ASR 和人声能力，不提供 Web 页面。

安装命令：

```bash
cd /opt/jusha-asr
sh jusha-asr-models-<版本>.run
```

安装后对外开放：

```text
http://<模型服务器IP>:9851  ASR 推理服务
http://<模型服务器IP>:9852  3D-Speaker 人声服务
```

如需调整 3D-Speaker 端口、GPU 或 Docker 网络，编辑：

```bash
cd /opt/jusha-asr/jusha-asr-models/jusha-asr-speaker
vi .env
```

常用配置：

```env
SA_PORT=9852
SA_GPU_ID=0
SA_DEVICE=cuda:0
SA_DOCKER_NETWORK_NAME=jusha-asr
SA_DOCKER_SUBNET=AUTO
```

保存后重新安装或升级该服务：

```bash
sh install.sh upgrade
```

ASR 推理服务的端口和 GPU 配置通常在交付包生成时已经固定。如现场必须调整，请联系交付或研发人员确认后再修改 `jusha-asr-asr/docker-compose.yml`。

## 8. 修改服务器 IP 或访问域名

### 8.1 只改变访问地址

如果服务器 IP 变化，或者需要改成域名访问，主要修改业务服务目录下的 `.env`。

```bash
cd /opt/jusha-asr/jusha-asr-business
cp .env .env.bak.$(date +%Y%m%d%H%M%S)
vi .env
```

将以下值改成新的 IP 或域名：

```env
ASR_TLS_COMMON_NAME=<新IP或域名>
ASR_TLS_ALT_NAMES=DNS:localhost,IP:127.0.0.1,IP:<新IP>
ASR_HTTP_PORT=9855
ASR_HTTPS_PORT=9856
```

如果使用域名，`ASR_TLS_ALT_NAMES` 示例：

```env
ASR_TLS_COMMON_NAME=asr.example.local
ASR_TLS_ALT_NAMES=DNS:localhost,DNS:asr.example.local,IP:127.0.0.1
```

如果使用自签名证书，并且旧证书里没有新 IP 或域名，需要删除旧证书后让系统重新生成：

```bash
rm -f runtime/certs/tls.crt runtime/certs/tls.key
sh install.sh upgrade
```

如果现场使用正式证书，请不要删除证书文件，直接将新的 `tls.crt` 和 `tls.key` 放到 `runtime/certs/` 后再执行：

```bash
sh install.sh upgrade
```

### 8.2 修改模型服务器地址

业务服务调用模型服务的地址在 `.env` 中配置：

```env
ASR_SERVICES_ASR=http://<模型服务器IP>:9851
ASR_SERVICES_SPEAKER_SERVICE_URL=http://<模型服务器IP>:9852
```

修改后执行：

```bash
sh install.sh upgrade
```

### 8.3 修改端口

业务入口端口：

```env
ASR_HTTP_PORT=9855
ASR_HTTPS_PORT=9856
```

3D-Speaker 端口：

```env
SA_PORT=9852
```

修改端口前请确认新端口未被占用，并同步调整防火墙策略。修改业务端口后，用户访问地址、客户端下载页地址也要同步变更。

### 8.4 桌面客户端默认地址

业务包通常会把 Windows 桌面客户端安装包放在 `runtime/downloads`，下载页会自动展示。桌面客户端安装包的默认服务器地址是在打包时写入的。

如果安装后服务器 IP 变化，已有客户端通常可以在客户端设置中手动修改服务器地址；如果希望下载页里的安装包默认地址也变成新 IP，需要重新获取按新 IP 打包的 `jusha-asr-business` 或 `jusha-asr-all` 交付包。

## 9. 配置文件和数据目录

### 9.1 业务服务目录

业务服务目录一般为：

```text
/opt/jusha-asr/jusha-asr-business
```

全量部署时为：

```text
/opt/jusha-asr/jusha-asr-all/jusha-asr-business
```

重要文件和目录：

| 路径 | 说明 |
| --- | --- |
| `.env` | 主要配置文件，IP、端口、管理员初始化、模型服务地址都在这里 |
| `docker-compose.yml` | 容器编排文件，一般不建议手工修改 |
| `runtime/mysql` | 数据库数据目录，生产环境必须备份 |
| `runtime/uploads` | 音频和上传文件目录，生产环境必须备份 |
| `runtime/certs` | HTTPS 证书目录 |
| `runtime/downloads` | 客户端安装包下载目录 |
| `runtime/term-catalog` | 影像术语库目录 |
| `backups` | 安装或升级时自动备份 `.env`、证书和 compose 文件 |

### 9.2 常用业务配置

| 配置项 | 说明 | 何时修改 |
| --- | --- | --- |
| `ASR_HTTP_PORT` | HTTP 访问端口 | 端口冲突或客户要求固定端口 |
| `ASR_HTTPS_PORT` | HTTPS 访问端口 | 端口冲突或客户要求固定端口 |
| `ASR_TLS_COMMON_NAME` | 证书主名称 | 修改服务器 IP 或域名 |
| `ASR_TLS_ALT_NAMES` | 证书可访问 IP 或域名列表 | 修改服务器 IP 或域名 |
| `ASR_SERVICES_ASR` | ASR 模型服务地址 | 模型服务分离部署或 IP 变化 |
| `ASR_SERVICES_SPEAKER_SERVICE_URL` | 3D-Speaker 人声服务地址 | 模型服务分离部署或 IP 变化 |
| `ASR_BOOTSTRAP_ADMIN_USERNAME` | 首次初始化管理员用户名 | 仅数据库第一次启动前有效 |
| `ASR_BOOTSTRAP_ADMIN_PASSWORD` | 首次初始化管理员密码 | 仅数据库第一次启动前有效 |
| `ASR_MYSQL_ROOT_PASSWORD` | MySQL root 密码 | 仅首次初始化前修改，初始化后不要随意改 |
| `ASR_DOCKER_NETWORK_NAME` | Docker 内部网络名 | 与现场已有 Docker 网络冲突时修改 |
| `ASR_DOCKER_SUBNET` | Docker 内部网段 | 自动选择失败或与现场路由冲突时指定 |

### 9.3 修改配置后的生效方式

业务服务修改 `.env` 后，优先执行：

```bash
cd /opt/jusha-asr/jusha-asr-business
sh install.sh upgrade
```

全量部署时路径改为：

```bash
cd /opt/jusha-asr/jusha-asr-all/jusha-asr-business
sh install.sh upgrade
```

`install.sh upgrade` 会自动备份当前 `.env` 和 `docker-compose.yml`，并重新创建容器。

## 10. 升级

### 10.1 升级前备份

升级前建议至少备份以下内容：

```bash
cd /opt/jusha-asr
tar czf backup-jusha-asr-$(date +%Y%m%d%H%M%S).tar.gz \
  jusha-asr-business/.env \
  jusha-asr-business/runtime/mysql \
  jusha-asr-business/runtime/uploads \
  jusha-asr-business/runtime/certs \
  jusha-asr-business/runtime/term-catalog
```

如果是全量部署，将路径替换为 `jusha-asr-all/jusha-asr-business/...`。

### 10.2 业务包升级

将新版本 `jusha-asr-business-<版本>.run` 和所有分包放到原安装父目录，执行：

```bash
cd /opt/jusha-asr
sh jusha-asr-business-<新版本>.run
```

升级会保留现有 `.env`、数据库、上传文件、术语库和证书，刷新镜像、脚本、`docker-compose.yml` 和下载页文件。升级过程中会自动备份配置到 `backups/<时间戳>`。

### 10.3 全量包升级

将新版本 `jusha-asr-all-<版本>.run` 和所有分包放到原安装父目录，执行：

```bash
cd /opt/jusha-asr
sh jusha-asr-all-<新版本>.run
```

升级顺序为模型服务后业务服务。升级后请重新验证登录页、下载页和一次音频识别任务。

### 10.4 模型包升级

```bash
cd /opt/jusha-asr
sh jusha-asr-models-<新版本>.run
```

模型包升级后，业务服务通常不需要重装；如果模型服务 IP、端口或网络名称变化，再同步修改业务 `.env`。

## 11. 启停、卸载和日志

### 11.1 查看状态

```bash
docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}'
```

常见容器名：

```text
jusha-asr-business
jusha-asr-asr
jusha-asr-speaker
```

### 11.2 查看日志

```bash
docker logs --tail 200 jusha-asr-business
docker logs --tail 200 jusha-asr-asr
docker logs --tail 200 jusha-asr-speaker
```

持续跟踪日志：

```bash
docker logs -f jusha-asr-business
```

### 11.3 重启服务

业务服务：

```bash
cd /opt/jusha-asr/jusha-asr-business
sh install.sh upgrade
```

全量部署下的业务服务：

```bash
cd /opt/jusha-asr/jusha-asr-all/jusha-asr-business
sh install.sh upgrade
```

3D-Speaker 服务：

```bash
cd /opt/jusha-asr/jusha-asr-models/jusha-asr-speaker
sh install.sh upgrade
```

### 11.4 普通卸载

普通卸载会停止并删除容器，保留数据和配置。

业务包：

```bash
cd /opt/jusha-asr/jusha-asr-business
sh uninstall.sh
```

模型包：

```bash
cd /opt/jusha-asr/jusha-asr-models
sh uninstall.sh
```

全量包：

```bash
cd /opt/jusha-asr/jusha-asr-all
sh uninstall.sh
```

### 11.5 清空数据卸载

清空数据不可恢复，执行前必须确认已经备份。

```bash
sh uninstall.sh purge
```

如需同时删除本地镜像：

```bash
sh uninstall.sh purge --remove-image
```

## 12. 防火墙和网络

如果服务器开启了防火墙，需要放行以下端口：

```text
9855/tcp  业务 HTTP
9856/tcp  业务 HTTPS
9851/tcp  ASR 推理服务，分离部署时业务服务器需要访问
9852/tcp  3D-Speaker 人声服务，分离部署时业务服务器需要访问
```

如果业务服务和模型服务在同一台服务器且使用全量包部署，通常不需要对外开放 `9851` 和 `9852`；但如果要从其他服务器调用模型服务，则需要放行。

Docker 内部网络默认名为 `jusha-asr`。安装脚本会自动选择不冲突的网段。如现场有特殊网段规划，可在 `.env` 中指定：

业务服务：

```env
ASR_DOCKER_NETWORK_NAME=jusha-asr
ASR_DOCKER_SUBNET=10.248.0.0/24
```

3D-Speaker：

```env
SA_DOCKER_NETWORK_NAME=jusha-asr
SA_DOCKER_SUBNET=10.248.10.0/24
```

Qwen3-ASR 推理服务如果需要手工指定网段，由交付人员在模型包部署时执行：

```bash
QWEN3_ASR_SUBNET=10.255.250.0/24 bash deploy.sh
```

## 13. 常见问题 QA

### Q1：执行 `.run` 提示找不到 `.part001` 文件

`.run` 只是安装入口，实际内容在 `.run.part001`、`.run.part002` 等分包里。请确认所有同名分包都和 `.run` 放在同一目录，并且文件名未被浏览器或传输工具改名。

### Q2：提示 `docker 未安装` 或 `未找到 docker compose`

服务器没有安装 Docker 或 Compose。先安装 Docker Engine 和 Docker Compose 插件，再重新执行安装。安装后确认当前用户可以执行：

```bash
docker info
docker compose version
```

### Q3：当前用户无法访问 Docker

通常是 Docker 服务未启动，或当前用户没有 Docker 权限。可先用 `sudo docker info` 验证。如果只有 `sudo` 能执行 Docker，请用具备权限的用户安装，或将当前用户加入 `docker` 用户组后重新登录。

### Q4：端口被占用怎么办

先查看占用：

```bash
ss -ltnp | grep -E ':9855|:9856|:9851|:9852'
```

如果业务端口冲突，修改业务 `.env`：

```env
ASR_HTTP_PORT=<新HTTP端口>
ASR_HTTPS_PORT=<新HTTPS端口>
```

然后执行：

```bash
sh install.sh upgrade
```

如果模型端口冲突，优先联系交付人员确认调整方式。

### Q5：浏览器提示证书不受信任

默认是自签名证书，属于内网部署的正常现象。可从以下地址下载证书并导入浏览器或操作系统信任区：

```text
https://<服务器IP>:9856/downloads/certs/tls.crt
```

也可以在服务器上获取：

```text
runtime/certs/tls.crt
```

### Q6：更换 IP 后浏览器仍提示证书不匹配

旧证书里没有新 IP。编辑 `.env` 的 `ASR_TLS_COMMON_NAME` 和 `ASR_TLS_ALT_NAMES` 后，删除旧自签名证书并升级：

```bash
rm -f runtime/certs/tls.crt runtime/certs/tls.key
sh install.sh upgrade
```

### Q7：改了 `.env` 中的管理员密码，登录密码没有变化

`ASR_BOOTSTRAP_ADMIN_PASSWORD` 只在数据库首次初始化时生效。数据库已经创建管理员后，修改 `.env` 不会覆盖已有密码。请登录系统后在用户管理中修改密码；如果是未投产测试环境，也可清空数据后重新初始化。

### Q8：改了 MySQL 密码后服务启动失败

`ASR_MYSQL_ROOT_PASSWORD` 会在首次初始化数据库时写入 `runtime/mysql`。数据库初始化后不要单独修改该值，否则旧数据目录里的密码和 `.env` 不一致会导致鉴权失败。如确需调整，请联系研发或交付人员处理。

### Q9：模型服务启动失败，日志提示 GPU 或 NVIDIA runtime 问题

在模型服务器执行：

```bash
nvidia-smi
docker run --rm --gpus all nvidia/cuda:12.2.0-base-ubuntu22.04 nvidia-smi
```

如果第一条失败，说明 NVIDIA 驱动未就绪；如果第二条失败，多数是 `nvidia-container-toolkit` 未安装或未配置。处理后重新执行模型包安装或 `sh install.sh upgrade`。

### Q10：业务页面能打开，但识别任务失败

优先检查业务服务是否能访问模型服务。分离部署时确认业务 `.env` 中：

```env
ASR_SERVICES_ASR=http://<模型服务器IP>:9851
ASR_SERVICES_SPEAKER_SERVICE_URL=http://<模型服务器IP>:9852
```

同时确认业务服务器到模型服务器的 `9851`、`9852` 端口网络可达，防火墙已放行。

### Q11：Docker 网络和现场内网冲突

安装脚本会自动避让常见冲突。如果仍然冲突，可在 `.env` 中指定一个现场未使用的 Docker 内部网段，例如：

```env
ASR_DOCKER_SUBNET=10.250.0.0/24
SA_DOCKER_SUBNET=10.250.10.0/24
```

保存后重新执行安装或升级。不要使用和客户办公网、VPN、服务器网卡同网段的地址。

### Q12：升级后服务健康检查失败

业务包升级失败时脚本会尝试回滚上一版镜像。请先保留现场，不要删除目录，收集以下信息交给研发或交付人员：

```bash
docker ps -a
docker logs --tail 300 jusha-asr-business
docker inspect jusha-asr-business
```

同时保留业务目录下 `backups/<时间戳>`。

### Q13：磁盘空间不足

模型包和 Docker 镜像较大。建议将安装目录放在空间充足的数据盘，例如 `/data/jusha-asr`。如 Docker 根目录空间不足，需要清理无用镜像或将 Docker 数据目录迁移到数据盘。

### Q14：下载页没有桌面客户端安装包

检查业务目录：

```bash
ls runtime/downloads
```

如果目录为空，说明该交付包没有打入客户端安装包，或升级时下载目录被清理。请将交付的 Windows 客户端安装包放入 `runtime/downloads`，或重新获取包含客户端安装包的业务交付包。

### Q15：是否可以直接编辑 `docker-compose.yml`

一般不建议。常见配置都应修改 `.env`。升级时 `docker-compose.yml` 可能会被新版本刷新，手工改动容易丢失。只有交付或研发人员明确要求时才编辑 `docker-compose.yml`。

## 14. 现场交付检查清单

部署完成后建议逐项确认：

- `docker ps` 中业务、ASR、人声服务容器状态正常。
- 浏览器可以打开 `https://<服务器IP>:9856/login`。
- 下载页 `https://<服务器IP>:9856/downloads` 可以打开。
- 管理员账号可以登录，并已修改默认密码。
- 上传测试音频后可以完成识别。
- 如启用说话人相关能力，3D-Speaker 健康检查 `http://<模型服务器IP>:9852/api/v1/health` 正常。
- 防火墙已按部署方式放行必要端口。
- 已记录安装目录、服务器 IP、端口、管理员账号保管方式和证书处理方式。
- 已备份 `.env`、证书和业务数据目录，或纳入现场备份策略。
