# 语音转写系统技术验证方案

> **文档版本**：v1.0
> **适用项目**：语音转写产品（局域网私有化部署版 · 基于 Qwen3-ASR）
> **文档目的**：在正式开发前，对核心技术链路逐项验证，识别技术风险，确认方案可行性

---

## 1. 验证目标与总体策略

### 1.1 验证目标

本方案聚焦以下核心技术风险的验证，确保在进入工程开发阶段前，关键技术路径已被打通且指标可量化：

- Qwen3-ASR 模型在目标硬件上的实际推理性能与精度
- 实时流式转写端到端延迟是否满足 < 2s 要求
- 不做模型微调的前提下，术语纠错管道的有效性
- 说话人分离模型的精度与 ASR 结果的对齐可行性
- 长音频分片拼接策略的质量损失
- NLP 摘要模型的生成质量与资源占用
- 系统整体并发能力

### 1.2 验证阶段划分

| 阶段 | 内容 | 周期 | 前置条件 |
|---|---|---|---|
| **阶段一：单模型能力验证** | ASR 模型部署、精度测试、流式延迟测试 | 第 1-2 周 | GPU 服务器到位 |
| **阶段二：纠错管道验证** | 三层纠错管道搭建与效果评估 | 第 2-3 周 | 阶段一完成 |
| **阶段三：会议场景验证** | 说话人分离、时间戳对齐、长音频分片 | 第 3-4 周 | 阶段一完成 |
| **阶段四：端到端集成验证** | 全链路延迟、并发压测、摘要质量 | 第 4-5 周 | 阶段一二三完成 |

## 3. 阶段一：ASR 模型能力验证

### 3.1 模型部署验证

**验证目标**：确认 Qwen3-ASR 模型可在目标硬件上通过 vLLM 正常启动并提供服务。

**验证步骤**：

1. 使用官方 Docker 镜像拉起 Qwen3-ASR-1.7B 推理服务
2. 确认 vLLM 服务正常监听，GPU 显存占用符合预期
3. 发送测试音频，确认返回转写结果
4. 分别验证流式（WebSocket）和非流式（HTTP）两种调用模式
5. 重复以上步骤验证 Qwen3-ASR-0.6B

**通过标准**：

- 模型加载成功，显存占用 ≤ 20GB（1.7B）/ ≤ 8GB（0.6B）
- 流式和非流式均可正常返回结果
- 服务连续运行 24 小时无崩溃

**参考命令**：

```bash
# 启动 ASR 推理服务（根据官方文档调整）
docker run --gpus all -p 8000:8000 \
  -v ./Qwen3-ASR-1.7B:/model \
  qwen3-asr:latest \
  --model /model --served-model-name qwen3-asr
```

### 3.2 识别精度验证

**验证目标**：量化模型在通用场景和专业场景下的识别错误率。

**验证方法**：

1. 对通用测试集跑全量转写，计算 CER（字错误率）
2. 对专业领域测试集跑全量转写，计算 CER 并统计术语错误分布
3. 对方言测试集跑全量转写，计算 WER（词错误率）

**评估脚本逻辑**：

```python
# CER 计算核心逻辑
import jiwer

# 加载标注文本和 ASR 输出
reference = load_reference("ground_truth.txt")
hypothesis = load_hypothesis("asr_output.txt")

# 计算 CER
cer = jiwer.cer(reference, hypothesis)
print(f"CER: {cer:.4f}")

# 术语级错误统计
term_errors = []
for ref_sent, hyp_sent in zip(reference, hypothesis):
    for term in medical_terms:
        if term in ref_sent and term not in hyp_sent:
            term_errors.append({
                "term": term,
                "reference": ref_sent,
                "hypothesis": hyp_sent
            })
```

**通过标准**：

| 场景 | 指标 | 目标值 | 红线值（不通过） |
|---|---|---|---|
| 通用普通话 | CER | ≤ 5% | > 8% |
| 专业领域（纠错前） | CER | ≤ 10% | > 15% |
| 方言 | WER | ≤ 16% | > 25% |

**额外产出**：术语错误 case 清单，直接作为纠错管道第一层词典的输入。

### 3.3 流式推理延迟验证

**验证目标**：确认流式转写链路的各环节延迟分布。

**验证方法**：

1. 客户端模拟程序每 300ms 发送一个音频 chunk（通过 WebSocket）
2. 记录每个 chunk 发送时间和首个识别结果返回时间
3. 连续测试 10 分钟音频，统计延迟分布（P50 / P95 / P99）

**埋点测量**：

```
T0: 客户端采集音频 chunk 完成
T1: chunk 通过 WebSocket 发出
T2: 服务端收到 chunk
T3: ASR 模型输出首个 token
T4: 服务端返回识别结果
T5: 客户端收到结果

需关注：
- 网络延迟 = T2 - T1
- 模型推理延迟（TTFT）= T3 - T2
- 端到端延迟 = T5 - T0
```

**通过标准**：

| 指标 | 目标值 | 红线值 |
|---|---|---|
| 端到端延迟 P50 | ≤ 500ms | > 1000ms |
| 端到端延迟 P95 | ≤ 1000ms | > 1500ms |
| 端到端延迟 P99 | ≤ 1500ms | > 2000ms |

## 4. 阶段二：术语纠错管道验证

### 4.1 验证前提

本阶段验证不做模型微调的前提下，纯后处理链路对术语识别的增强效果。纠错管道采用三层串行架构，应用于实时转写场景。

### 4.2 纠错管道架构

```
ASR 原始输出
    │
    ▼
[第一层] 精确词典替换
    │  - 数据结构：HashMap，key 为错误写法，value 为正确术语
    │  - 支持按科室/学科加载不同词典
    │  - 预期延迟：< 5ms
    │
    ▼
[第二层] 编辑距离模糊匹配
    │  - 算法：SymSpell（预构建删除编辑索引）
    │  - 编辑距离阈值：1-2
    │  - 预期延迟：< 10ms
    │
    ▼
[第三层] 拼音音近纠错
    │  - 工具：pypinyin 转拼音 + 拼音倒排索引
    │  - 匹配策略：拼音序列相似度 > 0.85 时触发替换
    │  - 预期延迟：< 10ms
    │
    ▼
纠错后输出（总延迟 < 25ms）
```

### 4.3 验证步骤

**步骤一：构建术语库与错误映射表**

1. 从阶段一的术语错误 case 清单中提取高频错误，构建第一层映射表
2. 将完整术语表导入 SymSpell 索引
3. 将术语表转拼音，构建拼音倒排索引

**步骤二：逐层效果评估**

对专业领域测试集，分别测量：

| 评估项 | 方法 |
|---|---|
| 基线（无纠错） | 直接使用阶段一的 ASR 原始输出，计算 CER |
| +第一层 | 仅开启词典替换，计算 CER |
| +第一层+第二层 | 叠加编辑距离，计算 CER |
| +三层全开 | 叠加拼音纠错，计算 CER |

同时统计每一层的：
- 纠正数（正确修正的术语数）
- 误纠数（把正确文本改错的数量）
- 漏纠数（应纠未纠的数量）

**步骤三：延迟测量**

对每一层单独计时，确认三层总延迟 < 25ms。测试文本长度覆盖短句（10 字）到长句（200 字）。

### 4.4 通过标准

| 指标 | 目标值 | 红线值 |
|---|---|---|
| 专业领域 CER（三层纠错后） | ≤ 8% | > 12% |
| 相比基线 CER 降幅 | ≥ 20% | < 10% |
| 误纠率（误纠数/总纠正数） | ≤ 5% | > 10% |
| 三层总延迟 | ≤ 25ms | > 50ms |

### 4.5 攒句策略验证

**验证目标**：确认"攒句再插入"策略在实时场景下的体验可接受性。

**验证方法**：

1. 实现简易断句检测逻辑（基于标点符号 + 静音检测 + 最大等待时长）
2. 模拟 10 分钟连续口述，记录每句从说完到插入的延迟
3. 由 3-5 名测试人员主观评价流畅度（1-5 分）

**断句策略参数**：

| 参数 | 初始值 | 说明 |
|---|---|---|
| 标点触发 | 句号、问号、感叹号 | 检测到句末标点立即输出 |
| 逗号触发 | 逗号 | 逗号后等待 300ms，无后续则输出 |
| 静音触发阈值 | 600ms | 连续静音超过此时长则输出当前缓冲 |
| 最大缓冲时长 | 3s | 即使未检测到断句信号，也强制输出 |

**通过标准**：

| 指标 | 目标值 |
|---|---|
| 句级延迟（说完到显示）P95 | ≤ 1.5s |
| 主观流畅度评分 | ≥ 3.5 / 5 |

## 5. 阶段三：会议场景验证

### 5.1 说话人分离验证

**验证目标**：确认 pyannote-audio 在目标会议场景下的分离精度。

**验证步骤**：

1. 部署 pyannote-audio 3.1（优先 GPU，备选 CPU）
2. 对多人会议测试集执行说话人分离
3. 计算 DER（Diarization Error Rate）
4. 将分离结果与 ASR 转写结果对齐，验证合并逻辑

**DER 计算**：

```python
from pyannote.metrics.diarization import DiarizationErrorRate
metric = DiarizationErrorRate()
der = metric(reference_annotation, hypothesis_annotation)
```

**通过标准**：

| 指标 | 目标值 | 红线值 |
|---|---|---|
| DER（3-4人会议） | ≤ 15% | > 25% |
| DER（5-6人会议） | ≤ 20% | > 30% |
| 说话人标签与转写对齐成功率 | ≥ 95% | < 90% |

**备选方案**：若 pyannote-audio 不达标，切换验证 3D-Speaker 或 WeSpeaker。

### 5.2 时间戳对齐验证

**验证目标**：确认 ForcedAligner 输出的时间戳精度满足"点击定位回听"的需求。

**验证步骤**：

1. 部署 Qwen3-ForcedAligner-0.6B
2. 对 ASR 转写结果执行时间戳对齐
3. 抽取 100 个时间戳标注点，人工核对偏差

**通过标准**：

| 指标 | 目标值 |
|---|---|
| 词级时间戳偏差（中位数） | ≤ 100ms |
| 句级时间戳偏差（中位数） | ≤ 200ms |
| ForcedAligner 处理速度（RTF） | ≤ 0.01 |

### 5.3 长音频分片拼接验证

**验证目标**：确认分片策略不会在边界处引入明显质量损失。

**验证方法**：

1. 准备 3 段 60 分钟以上音频，有完整人工标注
2. 分别测试以下分片策略：

| 策略 | 分片长度 | 重叠时长 | 拼接方式 |
|---|---|---|---|
| 策略 A | 20 分钟 | 无重叠 | 直接拼接 |
| 策略 B | 20 分钟 | 30 秒重叠 | LCS 文本去重 |
| 策略 C | 20 分钟 | 60 秒重叠 | LCS 文本去重 + ForcedAligner 时间戳裁剪 |

3. 对每种策略，重点检查分片边界处的文本质量（人工标注边界前后各 30 秒区域的 CER）

**通过标准**：

| 指标 | 目标值 |
|---|---|
| 边界区域 CER 相比非边界区域劣化 | ≤ 2 个百分点 |
| 重复/漏字率（边界处） | ≤ 1% |
| 最优策略确认 | 选出一种策略在质量和效率间取得最佳平衡 |

## 6. 阶段四：端到端集成验证

### 6.1 实时转写全链路验证

**验证目标**：打通"麦克风 → 客户端 → ASR → 纠错 → 光标插入"完整链路。

**验证步骤**：

1. 搭建最小可用客户端原型（可用 Python 脚本模拟，不需完整 UI）
2. 实现 WebSocket 音频流发送 → ASR 流式接收 → 三层纠错 → 攒句输出
3. 模拟光标位置插入（写入文本文件或简易编辑框）
4. 连续口述 10 分钟，记录全链路延迟与输出文本质量

**通过标准**：

| 指标 | 目标值 | 红线值 |
|---|---|---|
| 端到端延迟（句级，P95） | ≤ 1.5s | > 2s |
| 通用场景 CER | ≤ 5% | > 8% |
| 专业场景 CER（含纠错） | ≤ 8% | > 12% |
| 连续运行稳定性 | 30 分钟无中断 | 出现崩溃或长时间无响应 |

### 6.2 会议转写全链路验证

**验证目标**：打通"音频上传 → 分片 → ASR → 说话人分离 → 时间戳 → 纠错 → 摘要"完整流水线。

**验证步骤**：

1. 上传一段 60 分钟多人会议录音
2. 系统自动执行完整流水线
3. 检查输出：带说话人标签的时间戳逐字稿 + 结构化摘要

**通过标准**：

| 指标 | 目标值 |
|---|---|
| 60 分钟音频总处理时间 | ≤ 15 分钟 |
| 转写 CER | ≤ 6% |
| 说话人标注正确率 | ≥ 85% |
| 摘要覆盖率（关键决议/待办） | ≥ 80%（人工评估） |
| 摘要幻觉率（虚构内容） | ≤ 5%（人工评估） |

### 6.3 NLP 摘要质量验证

**验证目标**：确认摘要模型在会议纪要生成任务上的可用性。

**验证方法**：

1. 准备 5 段不同类型的会议转写文本（项目会、汇报会、讨论会等）
2. 分别使用 Qwen3-4B 和 Qwen3-8B 生成摘要
3. 人工评估四个维度（每项 1-5 分）：

| 维度 | 说明 |
|---|---|
| 完整性 | 是否覆盖了所有关键议题和决议 |
| 准确性 | 是否存在与原文矛盾的虚构内容 |
| 结构化 | 是否按"核心内容-待办事项-决议"等结构输出 |
| 可读性 | 语言是否通顺、适合直接作为会议纪要分发 |

**Prompt 模板验证**：

```
你是专业的会议纪要整理员。请根据以下会议转写原文，生成结构化会议纪要。

要求：
1. 提取核心讨论内容，按议题分组
2. 列出明确的决议事项
3. 列出待办事项（含责任人和截止时间，如原文有提及）
4. 不要添加原文中没有的信息
5. 如果某些信息不明确，标注"待确认"

会议转写原文：
{transcription_text}
```

**通过标准**：

| 指标 | 目标值 |
|---|---|
| 四维度综合评分 | ≥ 3.5 / 5 |
| 幻觉率 | ≤ 5% |
| 单次摘要生成时间（万字输入） | ≤ 60s |

### 6.4 并发压测

**验证目标**：确认系统在目标并发数下的稳定性和性能。

**验证方法**：

1. 使用 Locust 或自定义脚本模拟多路并发
2. 逐步增加并发数（5 → 10 → 20 → 50），观察延迟和错误率变化
3. 持续压测 30 分钟，记录 GPU 利用率、显存、推理延迟、队列积压

**通过标准（标准方案，单卡 A10）**：

| 并发数 | 流式延迟 P95 | 错误率 | GPU 利用率 |
|---|---|---|---|
| 10 路 | ≤ 1s | 0% | 记录基线 |
| 20 路 | ≤ 1.5s | 0% | 记录峰值 |
| 超出上限时 | 服务降级而非崩溃 | — | 队列积压有上限 |

## 7. 风险与应对预案

| 风险项 | 触发条件 | 应对预案 |
|---|---|---|
| ASR 精度不达标 | 通用 CER > 8% | 检查音频采样率和编码格式是否匹配模型要求；切换 0.6B 对比排查是否为模型加载问题 |
| 术语纠错管道误纠率高 | 误纠率 > 10% | 收紧编辑距离阈值至 1；拼音匹配提高相似度门槛至 0.9；对高频误纠词加入白名单 |
| 流式延迟超标 | P95 > 1.5s | 排查 vLLM 配置（batch size、prefill chunk size）；检查 GPU 是否存在资源争抢 |
| 说话人分离 DER 过高 | DER > 25% | 尝试调整 pyannote 的聚类参数；切换 3D-Speaker/WeSpeaker 备选方案 |
| 长音频分片边界质量差 | 边界 CER 劣化 > 5% | 增大重叠区域至 90 秒；引入 VAD 在静音点分片而非固定时长分片 |

### A. ASR 流式调用示例（WebSocket）

```python
import asyncio
import websockets
import wave
import time

async def test_streaming_asr(audio_path, ws_url):
    """流式 ASR 延迟测试"""
    with wave.open(audio_path, 'rb') as wf:
        chunk_size = int(wf.getframerate() * 0.3)  # 300ms per chunk
        
        async with websockets.connect(ws_url) as ws:
            latencies = []
            while True:
                data = wf.readframes(chunk_size)
                if not data:
                    break
                
                t_send = time.time()
                await ws.send(data)
                result = await ws.recv()
                t_recv = time.time()
                
                latencies.append(t_recv - t_send)
            
            # 输出延迟统计
            import numpy as np
            print(f"P50: {np.percentile(latencies, 50)*1000:.0f}ms")
            print(f"P95: {np.percentile(latencies, 95)*1000:.0f}ms")
            print(f"P99: {np.percentile(latencies, 99)*1000:.0f}ms")

asyncio.run(test_streaming_asr("test.wav", "ws://192.168.1.100:8000/ws"))
```

from symspellpy import SymSpell
from pypinyin import pinyin, Style
from typing import Dict, List, Tuple

class TermCorrectionPipeline:
    """三层术语纠错管道"""
    
    def __init__(self, term_dict_path: str, term_list_path: str):
        # 第一层：精确映射表
        self.exact_map: Dict[str, str] = self._load_exact_map(term_dict_path)
        
        # 第二层：SymSpell 模糊匹配
        self.sym_spell = SymSpell(max_dictionary_edit_distance=2)
        self._build_symspell_index(term_list_path)
        
        # 第三层：拼音索引
        self.pinyin_index: Dict[str, str] = self._build_pinyin_index(term_list_path)
    
    def correct(self, text: str) -> Tuple[str, dict]:
        """执行三层纠错，返回纠错后文本和纠错记录"""
        corrections = {"layer1": [], "layer2": [], "layer3": []}
        
        # 第一层：精确替换
        for wrong, right in self.exact_map.items():
            if wrong in text:
                text = text.replace(wrong, right)
                corrections["layer1"].append((wrong, right))
        
        # 第二层：编辑距离模糊匹配
        text, l2_corrections = self._symspell_correct(text)
        corrections["layer2"] = l2_corrections
        
        # 第三层：拼音纠错
        text, l3_corrections = self._pinyin_correct(text)
        corrections["layer3"] = l3_corrections
        
        return text, corrections
