# 02 — 架构与授权设计

## 1. 总体架构

v0.3 在产品规格书 §4.1 拓扑基础上新增"开放平台"子域，复用现有 gateway / asr-api / nlp-api / admin-api，不引入新的容器；新增逻辑通过 `admin-api` 内的 `openplatform` 包与 `gateway` 的鉴权中间件协同实现。

```
[ 旧调用方 ]            [ 新三方系统 ]                [ 巨鲨管理员 ]              [ 桌面 / Web 用户 ]
      │ 无鉴权             │ Bearer <openapi_token>      │ JWT                       │ JWT
      ▼                    ▼                             ▼                           ▼
   ┌──────────────────────────── Nginx + Gateway (10010) ────────────────────────────┐
   │  路径前缀路由（按从上到下优先级匹配，分流互斥）：                                │
   │   /api/legacy/*      → 直通      → legacy 适配层（asr-api/nlp-api）            │
   │   /api/upload, /api/recognize, /api/recognize/vad, /api/health,                 │
   │   /api/meeting/summary, /api/text/correct, /api/templates,                      │
   │   /api/audio/to_summary, /api/task/{id}                                         │
   │                       → 直通     → legacy 适配层（与上同一处理）                │
   │   /openapi/v1/*       → OpenAuth → 业务路由（asr-api/nlp-api/admin-api）       │
   │   /api/admin/*        → JwtAuth  → admin-api                                    │
   │   /api/asr,/api/meetings,/api/nlp,/ws/events → JwtAuth → asr-api / nlp-api     │
   └────────────────────────────────────────────────────────────────────────────────┘
                  │                          │                       │
                  ▼                          ▼                       ▼
           ┌────────────┐             ┌────────────┐         ┌────────────┐
           │  asr-api   │             │  nlp-api   │         │ admin-api  │
           │ + legacy   │             │            │         │ + openplat │
           │ + openapi  │             │            │         │            │
           └─────┬──────┘             └─────┬──────┘         └─────┬──────┘
                 ▼                          ▼                      ▼
                          MySQL（新增 openplatform 相关表）
```

### 1.1 责任拆分

| 模块 | 位置 | 职责 |
|---|---|---|
| OpenPlatform 管理 | `admin-api/internal/.../openplatform` | 应用注册、Token 签发与吊销、能力清单、调用日志查询 |
| OpenAuth 中间件 | `gateway` 与 `internal/interfaces/middleware` 共享 | 解析 Bearer Token → 校验签名、有效期、能力范围、应用启用状态 |
| Open API Handler | `asr-api/internal/interfaces/api/openapi/*` | 把开放 API 请求映射到既有 application service |
| Skill 注册中心 | `admin-api/internal/.../skill` | Skill CRUD、回调测试、健康探活 |
| Legacy 适配层 | `asr-api/internal/interfaces/api/legacy/*`（独立子包，不复用 OpenAuth 中间件，不依赖 OpenPlatform 数据模型） | 旧接口字段映射 → 转发新 service，外形保持不变 |
| 调用日志 | `infrastructure/openplatform/audit` | 写入数据库行（索引）+ 文件（正文）；**仅记录新平台调用** |
| Legacy 访问日志 | `infrastructure/legacy/access_log` | 普通文件日志 `runtime/legacy-access.log`，不入库、不统计到 `app_id` 维度 |

## 2. 授权链路

### 2.1 凭证模型

```
OpenApp (三方应用)
 ├── id                : uint64
 ├── app_id            : string  (公开标识，例如 `app_3kj2...`)
 ├── app_secret_hash   : string  (HMAC 派生密钥的哈希；明文仅创建时返回一次)
 ├── name              : string
 ├── owner_user_id     : uint64  (在管理端创建该应用的管理员)
 ├── status            : enum    (active / disabled / revoked)
 ├── default_workflow  : map<capability, workflow_id?>
 ├── allowed_caps      : []string  (勾选过的能力清单)
 ├── created_at / updated_at
 └── meta              : json    (备注、回调白名单等)
```

### 2.2 Token 签发

平台采用"长期 secret + 短期 access_token"双层方案，简化但不裸用 secret：

1. **创建应用**：管理端调用 `POST /api/admin/openplatform/apps`，返回 `{ app_id, app_secret }`。`app_secret` 仅此一次以明文返回。
2. **换取 access_token**：三方调用 `POST /openapi/v1/auth/token`，请求体 `{ app_id, app_secret }`，返回 `{ access_token, expires_in: 7200 }`；token 为 JWT，签名密钥 = `HMAC(platform_secret, app_id)`，载荷包含 `app_id`、`allowed_caps`、`exp`。
3. **业务调用**：所有 `/openapi/v1/*` 请求需附 `Authorization: Bearer <access_token>`。
4. **重置 secret**：管理端 `POST /api/admin/openplatform/apps/:id/rotate-secret`；旧 token 立即失效（失效校验：JWT 中带 `secret_version`，与库内不一致即拒）。

### 2.3 鉴权中间件（OpenAuth）规则

| 步骤 | 拒绝码 | 备注 |
|---|---|---|
| 缺少 / 格式错误 Authorization | 401 `ERR_OPEN_AUTH_MISSING` | |
| Token 解析失败 / 签名无效 | 401 `ERR_OPEN_AUTH_INVALID` | |
| Token 过期 | 401 `ERR_OPEN_AUTH_EXPIRED` | 带 `Retry-After-Sec` |
| `app_id` 未找到或 `status != active` | 403 `ERR_OPEN_APP_DISABLED` | |
| `secret_version` 与库内不一致 | 401 `ERR_OPEN_AUTH_REVOKED` | |
| 请求路径对应能力不在 `allowed_caps` | 403 `ERR_OPEN_CAP_DENIED` | 每条路由声明所需能力 |
| 限流（每应用 30 req/s 软限流，可后台配置） | 429 `ERR_OPEN_RATE_LIMITED` | 附 `Retry-After` |

### 2.4 兼容接口的鉴权（明确：完全不做鉴权）

`old http.py` 中所有路由原本无鉴权。v0.3 兼容层**保留这一现状**，理由：现有调用方已完成集成、不具备改造条件，强制鉴权会直接打断生产链路。

- 兼容层暴露两组等价路径（功能完全相同）：
  - `/api/legacy/*` —— 新规范化路径，便于内部识别 / 监控；
  - `/api/upload`、`/api/recognize`、`/api/recognize/vad`、`/api/health`、`/api/meeting/summary`、`/api/text/correct`、`/api/templates`、`/api/audio/to_summary`、`/api/task/{id}` —— 与旧版**逐字符**一致，存量调用方零改造。
- 这两组路径在 gateway 与各服务里**不挂任何鉴权中间件**：不解析 token、不读取 OpenAuth 上下文、不报 401 / 403。
- 与新平台的隔离手段（见 §3）保证"鉴权漏洞"不会蔓延到 `/openapi/v1/*` 与 `/api/admin/*`。

> 设计取舍：「未授权请求一律拒绝」的承诺**仅适用于新平台 `/openapi/v1/*`**。兼容接口属于明确的过渡资产，按既有契约放过；由部署侧的网络隔离（nginx 白名单 / 内网限制）控制风险，应用层不再"补刀"。

## 3. 与已有鉴权的隔离

每条请求**只走一种链路**，互斥分流：

| 路径前缀 | 鉴权 | 数据隔离 |
|---|---|---|
| `/api/admin/*` | JWT（管理员 / 业务用户） | admin-api 现有数据 |
| `/api/asr`、`/api/meetings`、`/api/nlp`、`/ws/events` | JWT（Web / 桌面端） | 既有业务表 |
| `/openapi/v1/*` | OpenAuth（Bearer access_token） | OpenPlatform 数据模型（`open_apps` / `open_skills` / `open_call_logs`） |
| `/api/legacy/*` 与旧 `/api/upload` 等 9 条路径 | **无鉴权** | 仅复用 ASR / NLP 引擎能力，不写 OpenPlatform 表，不与 `app_id` 关联 |

隔离要点：

- **路由互斥**：上述前缀按从精确到一般的顺序匹配；任意一条路径只能命中其中一组中间件链。
- **代码隔离**：legacy 适配层在独立子包（`internal/interfaces/api/legacy/`），不 import OpenAuth / `open_*` repo；编译期阻止误用。
- **数据隔离**：legacy 调用不带 `app_id`，不进入 `open_call_logs` / `skill_invocations`。日志另行写入 `runtime/legacy-access.log`。
- **可观测隔离**：Dashboard 的"按应用维度"指标不计 legacy；legacy 调用量由独立 widget 展示（来源 = 文件日志聚合或一张可选独立小表 `legacy_call_metrics`）。
- **能力范围隔离**：legacy 接口不暴露 `workflow_id`、不对接 Skill；即便底层 service 同源，参数被适配层截断。

## 4. 能力声明与路由绑定

每条开放路由在注册时声明所需能力，OpenAuth 中间件读取声明做判定：

| 能力 ID | 含义 | 触达的路由前缀 |
|---|---|---|
| `asr.recognize` | 语音转文字 | `/openapi/v1/asr/...` |
| `asr.stream` | 实时流识别 | `/openapi/v1/asr/stream-sessions/...` |
| `meeting.summary` | 会议纪要 | `/openapi/v1/meetings/...` |
| `nlp.correct` | 文本纠错（独立） | `/openapi/v1/nlp/correct` |
| `skill.register` | 注册 / 管理 skill | `/openapi/v1/skills/...` |
| `skill.invoke` | 接收语音指令回调（被调方权限） | 反向回调，注册时校验目标 URL |

应用勾选能力 → 写入 `OpenApp.allowed_caps` → token 派发时载入 → 中间件精准判断。

> 注意：兼容接口**没有**对应的 capability，也不通过 OpenApp 注册——它们独立于开放平台之外存在。

## 5. 调用日志

### 5.1 新平台调用（`/openapi/v1/*`）

- **行存**：`open_api_call_log(id, app_id, capability, route, status, latency_ms, request_id, ip, created_at)` —— 用于查询、聚合。
- **正文**：写入 `runtime/openapi-logs/<yyyy-mm-dd>/<request_id>.json`（请求摘要 + 响应 size），Dashboard 不展示但运维可下载。
- **保留**：默认 30 天，配置项 `openplatform.log_retention_days`。

### 5.2 兼容接口调用（`/api/legacy/*` 与旧 `/api/*`）

- **不入库**，避免污染 `open_call_logs` 与 `app_id` 维度统计。
- 仅写文本日志 `runtime/legacy-access.log`（标准 access log 格式 + `request_id`），由独立 logrotate 配置按天切割、保留 30 天。
- 无回调签名 / 无应用归属字段。
- 运维需要看趋势时，由 `scripts/legacy_log_metrics.sh` 离线聚合（v0.3 不强制做实时聚合）。

## 6. 与产品规格书 §4 的衔接

- 不破坏 ARCH-S* 的服务划分；新增逻辑限于 `admin-api` / `asr-api`。
- 不改变 gateway 是唯一对外入口的约束；`/openapi/v1` 与 `/api/legacy` 都需在 gateway 显式注册前缀。
- 不引入新的外部依赖；所有新增数据进 MySQL（见 [09-数据模型与迁移.md](09-数据模型与迁移.md)）。
