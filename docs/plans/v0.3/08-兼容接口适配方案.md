# 08 — 兼容接口适配方案

> 仅作为内部规划与下线评估使用，**不对外发布该文档内容**。
>
> **明确决策**：兼容接口**完全不做鉴权**。理由：现有调用方已完成集成、不具备改造条件，强加鉴权会直接打断生产链路。本节给出的是"功能 / 路由 / 数据 / 日志"的隔离方案，确保兼容接口与新平台互不污染。

## 1. 兼容范围（来自 `docs/old http.py`）

| 旧路径（保持不变） | 别名路径 | 处理方 | 说明 |
|---|---|---|---|
| `POST /api/upload` | `POST /api/legacy/upload` | asr-api | 上传 + 同步 / VAD 识别（按 `use_vad_segmentation` 字段） |
| `POST /api/recognize` | `POST /api/legacy/recognize` | asr-api | 同 `/api/upload` 但固定 `use_vad_segmentation=false` |
| `POST /api/recognize/vad` | `POST /api/legacy/recognize/vad` | asr-api | 同上但固定 `true` |
| `GET /api/health` | `GET /api/legacy/health` | gateway | 简单 200，沿用旧返回结构 |
| `POST /api/meeting/summary` | `POST /api/legacy/meeting/summary` | nlp-api | 文本→纪要 |
| `POST /api/text/correct` | `POST /api/legacy/text/correct` | nlp-api | 文本纠错 |
| `GET /api/templates` | `GET /api/legacy/templates` | nlp-api | 模板列表 |
| `POST /api/audio/to_summary` | `POST /api/legacy/audio/to_summary` | asr-api | 音频→纪要（含 `callback`） |
| `GET /api/task/{task_id}` | `GET /api/legacy/task/{task_id}` | asr-api | 旧任务状态 |

> 旧路径与别名路径**等价**（同 handler、同响应外形）；提供别名仅为内部识别 / 监控 / 后续单独下线。
> 公开文档不出现"别名路径"。

## 2. 鉴权约定（再次明确）

- **所有上表 18 条路径都不挂任何鉴权中间件**：不解析 Authorization 头、不返回 401/403。
- 即使请求里**带了** Authorization 也会被忽略（不是错误，但不参与判定）。
- 没有 `app_id`、没有能力检查、没有限流（如有限流需求由部署侧 nginx / WAF 承担）。
- gateway 路由表明确把这些路径绕过 OpenAuth 与 JwtAuth；新平台中间件不在该匹配链上注册。

## 3. 实现策略

- gateway：在最前置的路由组里直通 legacy 路径到 asr-api / nlp-api，不经过任何鉴权 filter。
- asr-api / nlp-api 内分别实现 `legacy_handler.go`，每个 handler 内部把请求转发到对应的新 service：
  - `legacy_upload_handler` → `asrService.RecognizeAudio` / `RecognizeWithVAD`
  - `legacy_audio_to_summary_handler` → `meetingService.ProcessAudioToSummary`（保留 callback 异步路径）
  - `legacy_summary_handler` → `nlpService.GenerateMeetingSummary`
  - `legacy_correct_handler` → `nlpService.ProcessASRResult`
  - `legacy_templates_handler` → `nlpService.ListTemplates`
  - `legacy_task_handler` → `taskService.GetTaskStatus`
- legacy 包**不 import** OpenAuth / OpenPlatform repo，编译期阻止误用。
- 路由注册分两组：
  - 旧路径 `/api/upload`、`/api/recognize`、`/api/recognize/vad`、`/api/health`、`/api/meeting/summary`、`/api/text/correct`、`/api/templates`、`/api/audio/to_summary`、`/api/task/:task_id`；
  - 别名 `/api/legacy/...`。
  两组指向同一 handler。

## 4. 字段映射规则

兼容层**保留旧响应外形**：成功响应里的 `success` / `message` / `data` / `partial_success` 等字段不变。新平台的标准响应格式（`code` + `data`）**不应用**到兼容接口。

| 旧字段 | 内部映射 | 备注 |
|---|---|---|
| `success: bool` | 由内部错误状态推算 | true = HTTP 2xx 且业务成功 |
| `message: str` | 错误信息或业务提示 | |
| `data` | 与旧逻辑一致 | 不附加任何新平台字段（如 `workflow_origin`），保持完全等价 |
| `partial_success` | 仅 `audio_to_summary` 路径 | 沿用旧语义 |
| `error_details` | 同上 | |
| `task_id` | 与新平台 task_id 共用同一格式 | |

## 5. 工作流附加在兼容接口上的行为

- **不接受 `workflow_id` 字段**（即使传入也忽略，不写日志）。兼容接口对接方升级到新接口才解锁工作流配置。
- 默认走 `asrService` / `meetingService` 的现有兜底链路，不查询 `OpenApp.default_workflows`（兼容请求没有 app_id，无法关联应用配置）。
- 如部署方希望兼容请求也跑工作流，可在 `legacy.default_workflow_id_for_*` 配置项中静态指定一个 system 工作流；默认空。

## 6. 数据与日志隔离

| 维度 | 新平台 | 兼容接口 |
|---|---|---|
| 调用日志 | `open_call_logs`（按 app_id） | `runtime/legacy-access.log`（普通文本） |
| Dashboard 指标 | 按 app_id 聚合 | 独立 widget，仅展示 QPS / 错误率 / 平均时延 |
| 数据库写入 | OpenPlatform 表 | 不写 OpenPlatform 任何表 |
| 错误码 | `ERR_OPEN_*` 体系 | 沿用旧 `success/message` |

兼容接口与 `open_apps`、`open_skills`、`open_call_logs` **零依赖**。

## 7. 下线节奏

| 阶段 | 时间窗口 | 动作 |
|---|---|---|
| Phase 0 | v0.3.0 GA | 兼容接口与新平台同时上线，对接方继续无感知调用 |
| Phase 1 | v0.3.0 GA + 1 月 | 兼容接口响应头开始下发 `Deprecation: 1` 与 `Sunset: <ISO 日期>`（仅添加头，不影响行为） |
| Phase 2 | v0.3.0 GA + 3 月 | 离线聚合 `legacy-access.log`，运营按 IP / UA 通知未迁移调用方 |
| Phase 3 | v0.4.0（预计 2026-Q4） | 在产品规格书 §13 中归档下线决议 |
| Phase 4 | v0.4.0 + 1 月 | gateway 配置开关 `legacy.enabled=false`，所有兼容路径返回 410 Gone（一次性切换） |
| Phase 5 | v0.5.0 | 删除 legacy 适配层代码与 access log 配置 |

每阶段在 Dashboard 的兼容 widget 上展示当前 QPS / 调用方 IP 数量，给运营提供决策依据。

## 8. 不暴露文档的执行细节

- `docs/openapi/`（新平台 SDK 用户文档）下**不**生成 legacy 章节。
- 内部 wiki 仅保留本文件作为映射表参考。
- 客服 / 售前回答对接方时统一答复：使用新接口的对应能力即可，旧接口不做新规格输出。

## 9. 测试要点（与 [11-验收清单.md](11-验收清单.md) 对齐）

1. 兼容路径**不带 token** 调用必须 200 / 业务正常（与 0.2.x 一致）。
2. 兼容路径带任意伪造 token 也能正常调用（token 被忽略）。
3. 响应字段与 0.2.x 录像逐字段一致。
4. `audio/to_summary` 的 callback 流程在新版下仍能正确触发。
5. gateway 关闭 `legacy.enabled` 后所有兼容路径返回 410。
6. legacy 调用不出现在 `open_call_logs`、`skill_invocations` 任何表中；新平台 Dashboard 数据不变。
7. 跨链路串扰用例：把同一 access_token 用到 legacy 路径不会"激活"任何 OpenAuth 上下文，反之 legacy 调用也不会污染 OpenAuth session。
